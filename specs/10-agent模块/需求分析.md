# Agent 模块 — 需求分析

> 对标参考：PaiSmart `ChatHandler.runReActLoop()`、`AgentToolRegistry`、`ChatGenerationStateService`

## 核心业务逻辑

将现有的"检索→Prompt→一次回答"线性链路，升级为 ReAct 多轮推理循环。LLM 通过 Function Calling 自主决定是否调用工具（检索知识库、生成摘要、查看统计），每轮工具结果回注 messages 驱动下一轮决策，最大 4 轮迭代 / 8 次工具调用后强制给出最终回答。

## 关键功能模块

### 1. ReAct 循环引擎 (P0)

- **职责**：管理 LLM ↔ Tool 的多轮交互，控制迭代次数和工具调用上限
- **入口**：`ChatHandler.processMessage()` → `runReActLoop()`
- **循环终止条件**：
  - LLM 返回 `finish_reason=stop` 且无 tool_calls → 正常结束
  - 达到 max_rounds (4) → 注入"预算已用尽"提示，最后一轮禁止工具调用
  - 达到 max_tool_calls (8) → 后续工具调用返回"预算已用尽"
  - 用户取消 → `cancelledGenerations` 标记
  - LLM 调用超时 (120s) → 断开连接并报错
- **上下文压缩**：历史消息仅保留最近 6 条，单条截断 800 字符

### 2. Tool 注册中心 (P0)

- **职责**：管理 Agent 可调用工具的定义和执行分发
- **注册格式**：OpenAI Function Calling 兼容
  ```json
  {
    "type": "function",
    "function": {
      "name": "search_knowledge",
      "description": "在知识库中检索与查询相关的内容片段",
      "parameters": {
        "type": "object",
        "properties": {
          "query": { "type": "string", "description": "检索关键词" }
        },
        "required": ["query"]
      }
    }
  }
  ```
- **首批工具**（3 个）：
  - `search_knowledge` → 调用 `RetrievalService.search()`，结果注入引用映射
  - `generate_summary` → 检索后二次调用 LLM 生成摘要，token 流式传到前端
  - `knowledge_stats` → 查询 ES 索引统计 + MySQL 文档计数

### 3. 工具 Schema 生成器 (P0)

- **职责**：用 `LinkedHashMap` 构建 OpenAI 兼容的 JSON Schema
- **要求**：每个工具的 name / description / parameters(含 type + required) 三要素完整
- **轻量实现**：不用 JSON Schema 库，直接 Map 序列化

### 4. 工具调用预算控制 (P1)

- **职责**：防止模型陷入工具调用死循环
- **规则**：
  - 全局 `max_tool_calls = 8`
  - 超过预算时工具不再执行，注入 `tool` role 消息："工具调用预算已用尽，请基于已有结果给出最终回答"
  - 第 4 轮开始时移除 tools 参数，强制模型直接回答

### 5. 生成任务生命周期管理 (P1)

- **职责**：追踪每次 LLM 生成的完整状态
- **Redis Key**：
  - `chat:generation:{id}:meta` → `{userId, conversationId, status, createdAt}`
  - `chat:generation:{id}:content` → 累积的响应文本
  - `chat:generation:{id}:refs` → `Map<引用编号, ReferenceInfo>` JSON
- **状态机**：`STREAMING → COMPLETED / FAILED / CANCELLED`
- **TTL**：30 分钟

### 6. 流式取消与资源回收 (P1)

- **职责**：用户点击停止时立即释放上游 LLM 连接
- **方案**：用 `WebClient` (WebFlux) 替代 `OkHttp`，返回 `Disposable`，取消时 `dispose()`
- **配额结算**：取消前结算已消耗的 Token

### 7. 工具调用状态推送 (P1)

- **职责**：前端实时展示 Agent 的工具执行过程
- **方案**：现有 SSE 通道增加结构化事件类型
  ```
  event: tool_call
  data: {"type":"tool_call","tool":"search_knowledge","status":"executing","generationId":"..."}
  ```
- **状态枚举**：`executing` → `success` / `failed`

## 边界条件

| 场景 | 处理策略 |
|------|----------|
| LLM 返回空 response | `finishCancelledGeneration()` 兜底，标记 FAILED |
| LLM 返回 tool_calls 但参数 JSON 解析失败 | catch 异常 → tool 执行失败 → 错误信息回注 messages |
| 用户消息触发无限检索循环 | max_rounds=4 + max_tool_calls=8，超限后强制回答 |
| 生成过程中 WebSocket/SSE 断开 | `cancelledGenerations` 标记，上游流取消 |
| 同一用户并发多条消息 | `chatMonitorExecutor` 线程池隔离，每用户一个 generation 排队 |
| ReAct 上下文超长 | 历史截断 6 条 × 800 字符，系统 prompt 不含检索结果占位 |
| Tool 执行中用户点停止 | `cancelledGenerations.add()`，工具执行完成后判断取消标记，不继续循环 |
| `generate_summary` 内部 LLM 调用失败 | 标记 `streamedToUser=true`，错误信息注入最终消息 |
| 工具返回结果超大 | 单条 tool message content 截断到 `MAX_CONTEXT_SNIPPET_LEN` |

## 潜在技术难点

**1. 流式 tool_calls 增量解析**
- 难点：OpenAI 流式返回中 tool_calls 是多 chunk 到达的（name/arguments 分多次 delta），需在客户端累加重组
- 方案：`ReActStreamAccumulator` 内部维护 `Map<Integer, StreamingToolCall>`，按 index 聚合 delta 碎片，流结束后统一解析

**2. 工具调用与流式输出的时序协调**
- 难点：`generate_summary` 工具内部调用 LLM 产生流式 token，需同时维持外层 ReAct 循环的消息格式完整性
- 方案：工具返回 `streamedToUser=true` → 外层跳过后续轮次直接 finalize

**3. 取消的竞态条件**
- 难点：在 `streamReActTurn` 阻塞等待期间收到取消信号，`responseFuture.get()` 可能已完成
- 方案：短轮询（200ms）+ `isGenerationCancelled()` 检查，配合 `CompletableFuture.completeExceptionally`

**4. OkHttp → WebClient 迁移**
- 难点：当前 DeepSeekLlmService 全面使用 OkHttp 同步/异步调用，迁移涉及流式解析和错误处理
- 方案：仅在 `LlmProviderRouter`（新文件）使用 WebClient，旧代码不碰

**5. Token 消耗估算**
- 难点：流式响应中 prompt_tokens / completion_tokens 在最后一帧才返回
- 方案：先预占（估算值），流结束后用 API 返回的实际值结算

### Stop Generation

```
POST /api/llm/stop
Body: {"generationId": "uuid"}  // 可选，不传则停止当前用户活跃生成

Response: {"code":200,"data":{"generationId":"uuid","status":"cancelled"}}
```

## 数据结构

### Redis: 生成状态

```
Key:   chat:generation:{id}:meta
Type:  Hash
Fields: userId, conversationId, status(STREAMING|COMPLETED|FAILED|CANCELLED), createdAt
TTL:   1800 (30min)

Key:   chat:generation:{id}:content
Type:  String (累积文本)
TTL:   1800

Key:   chat:generation:{id}:refs
Type:  String (JSON: {"1":{"fileMd5":"...","fileName":"...","pageNumber":5,...}})
TTL:   1800
```

### Redis: 用户反馈

```
Key:   feedback:{userId}
Type:  Hash
Fields: {timestamp} → "good:回答很准确" | "bad:信息来源不对"
过期策略：最近5条有效，旧的自然淘汰
```

## Agent 模块扩展规划

Agent 模块位于 llm 与 api 之间，架构位置：

```
common → storage → messaging → document → embedding → vector-store → retrieval → llm → agent → api
                                                                                      ↑ 新增
```

**分阶段实施**：

| 阶段 | 功能 | 依赖 |
|------|------|------|
| P0 | ReAct 循环 + Tool Registry + search_knowledge | retrieval |
| P0 | Tool Schema 生成器 | — |
| P1 | 工具调用预算控制 | P0 ReAct 循环 |
| P1 | 生成任务生命周期管理 | Redis |
| P1 | 流式取消与资源回收 | WebClient 迁移 |
| P1 | 工具调用状态推送(SSE 增强) | P0 ReAct 循环 |
| P1 | 反馈收集与自适应 | P0 Tool Registry |
