# Storage 模块 — 需求分析

## 核心业务逻辑

大文件分片上传到 MinIO，服务端合并。**Redis 记录上传进度，key=md5，支持断点续传。** 文件标识用 MD5，重复文件秒传。

### 完整流程

```

  │  1. 计算文件MD5            │                          │                     │
  │                           │                          │                     │
  │  2. InitUpload({md5,      │                          │                     │
  │     fileName, fileSize})  │                          │                     │
  │ ────────────────────────→ │  3. EXISTS upload:{md5}   │                     │
  │                           │ ───────────────────────→ │                     │
  │                           │ ←── 不存在               │                     │

  │                           │  4. createMultipartUpload                       │
```

### 断点续传流程

```
  网络中断，已上传第 1,3,5,7 片

  │  InitUpload({md5})                                  │
  │ ──────────────────────────────────────────→           │
  │                           │  EXISTS upload:{md5}      │
  │                           │  状态 = UPLOADING          │
  │                           │  ← 返回已有 uploadId       │
  │                           │     + 已上传 partNumbers   │
  │                           │                           │
  │  {status=RESUME, uploadId,│                           │
  │   uploadedParts:[1,3,5,7]}│                           │
  │ ←─────────────────────────                           │
  │                                                       │
  │  客户端只上传缺失的分片 [2,4,6,8,9,10...]                │
  │                                                       │
  │  CompleteUpload({md5}) → MinIO 合并 → 完成             │
```

## 关键功能模块

### 1. 上传初始化（InitUpload）
- 入参：`{fileMd5, fileName, fileSize}`
- **秒传**：Redis `EXISTS upload:{md5}` → 不存在 → 查 MySQL 元数据 → 存在且 READY → 返回 `status=EXIST`
- **断点续传**：Redis `EXISTS upload:{md5}` → 存在且状态 UPLOADING → 返回已有 uploadId + `HGETALL upload:{md5}:parts` 的 key 列表
- **新上传**：调 MinIO `createMultipartUpload` → Redis `HSET upload:{md5}` → 返回 uploadId

### 2. 分片上传（UploadPart）
- 入参：`md5, partNumber, MultipartFile`
- 调 MinIO `uploadPart` → 返回 etag
- Redis `HSET upload:{md5}:parts {partNumber} {etag}`
- 允许乱序，重复上传覆盖

### 3. 合并完成（CompleteUpload）
- 入参：`md5`
- Redis `HGETALL upload:{md5}:parts` → 按 partNumber 排序
- 校验：1 到 totalParts 连续无跳号
- 调 MinIO `completeMultipartUpload`
- Redis `DEL upload:{md5}` `DEL upload:{md5}:parts`
- MySQL 写入文件元数据

### 4. 进度查询（GetProgress）
- 入参：`md5`
- 返回：`{totalParts, uploadedPartNumbers[], uploadedSize, percentage}`

### 5. 取消上传（AbortUpload）
- 入参：`md5`
- MinIO `abortMultipartUpload`
- Redis DEL 两个 key

### 6. 小文件直传（DirectUpload）
- 入参：`MultipartFile`
- 服务端流式计算 MD5 → MinIO `putObject`
- MySQL 写入元数据 → 返回 fileId

### 7. 文件下载（Download）
- 入参：`fileId (md5)`
- MinIO 预签名 URL → 302

## 边界条件

| 场景 | 处理策略 |
|------|----------|
| 分片乱序 | 允许，Redis HSET 后提交时按 partNumber 排序 |
| 同一分片重复 | Redis 覆盖旧 etag |
| 断点续传 | InitUpload 时 Redis 已存在 UPLOADING → 返回已有 uploadId + 已上传分片列表 |
| 秒传 | Redis + MySQL 均无记录 → 新上传；MySQL 有 READY → 秒传 |
| 上传完成后再 InitUpload | Redis 已清理，MySQL 已有 READY → 秒传 |
| Redis key 过期 | TTL 24h，过期后通过 `Keyspace Notifications` 触发 MinIO abort |
| MinIO uploadId 过期 | 定时任务对比 Redis 与 MinIO，清理孤儿 upload |
| 并发上传相同 MD5 | Redis `HSETNX` 原子操作，先到先得，后来者返回已有 uploadId |
| 空文件 | 拒绝（size=0） |
| 单文件 > 100GB | `partSize = max(5MB, fileSize/10000)`，分片数上限 10000 |

## 潜在技术难点

**1. Redis ↔ MinIO 状态一致性**
- Redis TTL 24h，短于 MinIO uploadId 有效期（15天）
- Redis key 过期时通过 `Redis Keyspace Notifications` 回调 abort MinIO multipart
- 兜底：定时任务每小时 `listMultipartUploads` 对比 Redis，孤儿执行 abort

**2. 并发 InitUpload 同一 MD5**
- 用 Redis `HSETNX upload:{md5} uploadId xxx` 原子抢占
- 抢不到的 → `HGETALL` 返回已有 uploadId → 进入断点续传模式

**3. MinIO ETag 完整文件校验**
- CompleteUpload 后 MinIO 返回的完整对象 ETag 即为文件 MD5
- 与客户端上报的 fileMd5 比对，不一致则 abort + 告警

## Redis 数据结构

```
# 上传任务
Key:   upload:{md5}
Type:  Hash
Fields:
  md5        → d41d8cd...
  fileName   → 项目文档.pdf
  fileSize   → 104857600
  uploadId   → minio-uuid
  bucket     → localrag-documents
  objectKey  → d41d8cd.../项目文档.pdf
  totalParts → 20
  status     → UPLOADING
  createdAt  → 2026-05-05T15:00:00
TTL: 86400 (24h)

# 已上传分片
Key:   upload:{md5}:parts
Type:  Hash
Fields:
  1  → "etag1..."
  2  → "etag2..."
  5  → "etag5..."
TTL: 86400 (与父 key 同步)
```

# MySQL 持久化 — 需求分析

## 背景

当前 FileMetadata 存储在 `InMemoryFileMetadataRepository`（ConcurrentHashMap），应用重启数据全部丢失。需要接入 MySQL，确保文件元数据持久化。

## 核心逻辑

用 Spring Data JPA 替代内存 Map。file_metadata 表以 md5 为主键。

### 去重规则调整

| 场景 | 旧逻辑 | 新逻辑 |
|------|--------|--------|
| 相同 MD5 + 相同文件名 | 秒传 | 秒传（不变） |
| 相同 MD5 + **不同文件名** | 秒传（返回旧文件名） | **覆盖更新**：旧 MinIO 对象删除，上传新对象，更新 MySQL 记录 |
| 不同 MD5 | 新上传 | 新上传（不变） |

这样用户用不同文件名上传相同内容的文件，元数据会更新为新文件名。

## 关键决策

| 决策点 | 方案 |
|--------|------|
| ORM | Spring Data JPA + Hibernate |
| DDL | `spring.jpa.hibernate.ddl-auto=update` 自动建表 |
| 主键 | md5（VARCHAR 32） |
| 连接池 | HikariCP（Spring Boot 默认） |
| 数据库名 | `localrag`（手动创建或 `createDatabaseIfNotExist=true`） |
| 替代旧实现 | `InMemoryFileMetadataRepository` → `JpaFileMetadataRepository` |

## 表结构

```sql
CREATE TABLE file_metadata (
    md5          VARCHAR(32)  PRIMARY KEY,
    file_name    VARCHAR(500) NOT NULL,
    file_size    BIGINT       NOT NULL,
    object_key   VARCHAR(600) NOT NULL,
    bucket       VARCHAR(100) NOT NULL,
    content_type VARCHAR(200),
    status       VARCHAR(20)  NOT NULL DEFAULT 'READY',
    created_at   DATETIME     NOT NULL
);
```
