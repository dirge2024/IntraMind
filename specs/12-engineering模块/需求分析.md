# 系统保障与可观测性 — 需求分析

> 对标参考：PaiSmart `RateLimitService`、`UsageQuotaService`、`AiProperties`、`LlmProviderRouter`、`ChatHandler` 线程池隔离

## 一、频率限制与配额管理

### 核心业务逻辑

用 Redis 实现多级限流（按用户 / 按 IP / 全局），防止 API 滥用和控制 LLM Token 成本。聊天消息每分钟限频，Embedding/LLM 调用按天配额管控，超限返回 429。

### 限流流程

```
Client                   RateLimitService              Redis
  │  1. 请求任意 API        │                           │
  │ ─────────────────────→ │  2. checkChatByUser(uid)   │
  │                         │ ───────────────────────→ │
  │                         │  3. INCR chat:rate:user:{uid}:{minute}
  │                         │     EXPIRE 60
  │                         │ ←── count                 │
  │                         │                           │
  │                         │  4. count > limit?         │
  │  5. 429 Too Many Req    │     → throw RateLimitExceededException
  │     {retryAfterSeconds} │     → 正常处理            │
  │ ←────────────────────── │                           │
```

### Token 预算流程

```
ChatHandler              UsageQuotaService             Redis
  │  1. 发送前              │                           │
  │ ─────────────────────→ │  2. reserveLlmUsage()      │
  │                         │ ───────────────────────→ │
  │                         │  3. INCRBY llm:usage:user:{id}:{date} {estimatedTokens}
  │                         │     INCRBY llm:usage:global:{date} {estimatedTokens}
  │                         │ ←── currentTotal          │
  │                         │                           │
  │                         │  4. 超出日预算?            │
  │                         │     → 拒绝请求            │
  │                         │                            │
  │  5. LLM 流式响应结束     │                            │
  │ ─────────────────────→ │  6. settleReservation()    │
  │                         │ ───────────────────────→ │
  │                         │  7. 用实际 token 修正差值   │
  │                         │     INCRBY ... -{差额}     │
```

### 关键功能模块

#### 1. 用户级 + 全局级限流 (P0)

| 维度 | Key 设计 | 限制 | TTL |
|------|---------|------|-----|
| 用户聊天 | `chat:rate:user:{userId}:{minute}` | 10次/分钟 | 60s |
| 全局聊天 | `chat:rate:global:{minute}` | 100次/分钟 | 60s |
| 用户 LLM | `llm:rate:user:{userId}:{minute}` | 5次/分钟 | 60s |

#### 2. Token 日预算 (P1)

- **预算粒度**：每人每天 LLM Token 上限（如 100k tokens）
- **估算策略**：发送前按字符数估算（ASCII × 0.3, CJK × 0.95），响应后用 API 返回的实际值结算
- **全局预算**：每分钟系统级 Token 上限防溢出
- **存储**：`llm:usage:user:{id}:{date}` String，`INCRBY` 原子累加

#### 3. 超额处理

- 返回 `429 Too Many Requests`
- 响应体含 `retryAfterSeconds` 字段，前端据此倒计时

### 边界条件

| 场景 | 处理 |
|------|------|
| Redis 不可用 | 降级为无限制（打日志告警），不阻塞正常请求 |
| 跨分钟边界 | 按 `System.currentTimeMillis()/60000` 取整到分钟 |
| 日期变更 | Key 含 `LocalDate`，自然切换 |
| 预占与实际偏差 | settle 时 `INCRBY` 负差值修正，最少扣 0 |
| 预估 Token 超过剩余预算 | 请求直接拒绝，不预占 |

## 二、LLM Provider 路由器

### 核心业务逻辑

LLM 模块参照 Embedding 模块的双 Provider 设计，抽象统一接口，支持 DeepSeek / Qwen 双实现，用 `@ConditionalOnProperty` 切换。后续升级为 DB 配置驱动（`model_provider_config` 表），管理页切换无需重启。

### 接口抽象

```
LlmProvider (contract)
  ├── StreamHandle streamChat(userId, messages, onChunk, onError, onComplete)
  ├── ReActTurn streamReActTurn(userId, messages, tools, maxTokens, onChunk, onError, onComplete)
  └── void cancel(StreamHandle)

  Impl:
  ├── DeepSeekLlmProvider (当前代码提取)
  └── QwenLlmProvider (新增)
```

### 关键决策

| 决策点 | 方案 |
|--------|------|
| 通信库 | 暂时保持 OkHttp（DeepSeek），Qwen 可用 WebClient |
| 切换方式 | 先用 `@ConditionalOnProperty(localrag.llm.provider=deepseek/qwen)` |
| 管理页切换 | P2 阶段改为 DB 驱动，参考 PaiSmart ModelProviderConfigService |
| API Key 安全 | 环境变量注入，DB 驱动后用 AES 加密存储 |

## 三、可配置 Prompt

### 核心逻辑

将 `PromptBuilder` 硬编码的系统提示词抽到 `application.yml`，支持 `rules`、`noResultText`、`refStart`/`refEnd` 引用占位符。调整 Agent 行为不改代码。

### 配置结构

```yaml
localrag:
  prompt:
    rules: |
      你是IntraMind知识库AI助手「茉莉」。
      严格基于提供的参考资料回答用户问题。
      如果资料不足以回答问题，明确说明"知识库暂无相关信息"。
    refStart: "<<REF>>"
    refEnd: "<<END>>"
    noResultText: "（本轮无检索结果）"
```

### Prompt 组装逻辑

```
systemPrompt = rules
             + "\n"
             + refStart + "\n"
             + (检索结果拼接 | noResultText) + "\n"
             + refEnd

完整 messages = [system: systemPrompt]
              + history(最近6条)
              + [user: 当前问题]
```

## 四、线程池隔离

### 核心逻辑

LLM 流式调用耗时 10s-120s，不应阻塞 Tomcat/WebSocket 线程。将 ReAct 循环提交到独立线程池，主线程立即返回。

### 线程池配置

```java
@Bean("chatExecutor")
ThreadPoolTaskExecutor chatExecutor() {
    // core=4, max=8, queueCapacity=50
    // rejectedPolicy=CallerRunsPolicy（满后由调用线程执行）
}
```

### 隔离效果

- Tomcat 线程：接收 HTTP 请求 → 提交任务到 chatExecutor → 立即释放
- chatExecutor 线程：执行 ReAct 循环 → 流式写 SSE → 超时/完成

## 五、链路日志与可观测性

### 核心逻辑

每次对话生成唯一 `generationId`，通过 MDC 注入日志，贯穿 Agent → LLM → Tool 全链路。便于排查多轮推理问题。

### generationId 生命周期

```
生成: chatGenerationStateService.createGeneration()
注入: MDC.put("generationId", id)
清除: MDC.remove("generationId")  // finally 块
```

### Logback 配置

```xml
<pattern>%d{HH:mm:ss} [%thread] %-5level %logger{36} - [%X{generationId}] %msg%n</pattern>
```

### 关键日志点

| 节点 | 日志内容 |
|------|---------|
| 生成开始 | `generationId={id}, userId={uid}, conversationId={cid}` |
| ReAct 每轮 | `round={n}, toolCalls=[...], promptTokens={p}, completionTokens={c}` |
| Tool 执行 | `tool={name}, status=success/failed, elapsed={ms}` |
| 生成结束 | `finishReason={reason}, totalPromptTokens={p}, totalCompletionTokens={c}, elapsed={ms}` |
| 取消 | `generationId={id}, cancelled after round={n}` |

## 六、异常兜底策略

### 检索降级

```
向量 Embedding 成功 → KNN + BM25 混合检索
向量 Embedding 失败 → BM25 only（打日志）
ES 不可用       → 返回 503 + 明确提示
```

### LLM 降级

```
LLM 超时(120s)  → 返回已生成的部分内容 + "生成超时"
LLM API 返回 429 → 重试 1 次 → 仍失败返回 "服务繁忙"
LLM API 返回 5xx → 重试 1 次 → 仍失败返回 "AI 服务暂不可用"
```

### 文件处理降级

```
Tika 解析失败 → FileMetadata.status = PARSE_FAILED，支持手动重试
向量化失败   → FileMetadata.status = EMBED_FAILED，支持重索引 API
```

## 数据结构汇总

### Redis: 限流

```
Key:   chat:rate:user:{userId}:{minuteSlot}
Type:  String
Value: 当前分钟请求次数
TTL:   60s

Key:   chat:rate:global:{minuteSlot}
Type:  String
Value: 当前分钟全局请求次数
TTL:   60s
```

### Redis: Token 配额

```
Key:   llm:usage:user:{userId}:{localDate}
Type:  String
Value: 今日已消耗 Token 数
TTL:   到次日 0 点 (计算剩余秒数)

Key:   llm:usage:global:{localDate}
Type:  String
Value: 今日全局已消耗 Token 数
TTL:   到次日 0 点
```

### MySQL: model_provider_config (P2)

```sql
CREATE TABLE model_provider_config (
    id                  BIGINT AUTO_INCREMENT PRIMARY KEY,
    config_scope         VARCHAR(20)  NOT NULL,  -- 'llm' / 'embedding'
    provider_code        VARCHAR(30)  NOT NULL,  -- 'deepseek' / 'qwen' / 'zhipu'
    display_name         VARCHAR(50),
    api_style            VARCHAR(30)  NOT NULL DEFAULT 'openai-compatible',
    api_base_url         VARCHAR(300) NOT NULL,
    model_name           VARCHAR(100) NOT NULL,
    api_key_ciphertext   TEXT,                   -- AES 加密
    embedding_dimension  INT,
    is_enabled           BOOLEAN DEFAULT TRUE,
    is_active            BOOLEAN DEFAULT FALSE,
    updated_by           VARCHAR(50),
    updated_at           DATETIME,
    UNIQUE KEY uk_scope_provider (config_scope, provider_code)
);
```
