# Vector-Store 模块 — 需求分析

## 核心业务逻辑

消费 Kafka `embedding.requested` 消息，将 chunk（文本 + 2048 维向量）写入 Elasticsearch。入链在此闭合——**文件上传到 ES 存储全链路打通**。

### 入链全貌（本模块完成后）

```
上传文件 → MinIO 存储 → Kafka(document.uploaded)
  → Tika 解析 → 三级降级分块 → Kafka(document.chunked)
  → 通义千问/DeepSeek 向量化 → Kafka(embedding.requested)
  → ES 存储(text + dense_vector[2048])  ← 本模块
  ✓ 入链完毕，可端到端测试
```

### 关键决策

| 决策点 | 方案 | 理由 |
|--------|------|------|
| ES 客户端 | `elasticsearch-java` 8.10.0（官方新版） | 已在父 POM 中管理，与本地 ES 实例版本精确匹配 |
| 索引策略 | 单索引 `localrag-chunks` | 数据量不大时单索引够用，后续可按知识库分片 |
| text 映射 | `text` + `ik_max_word` 分词器 | 中文检索需要 IK 分词，Docker 已预装 |
| 向量映射 | `dense_vector`，dims=2048，index=true | 启用 KNN 索引 |
| 相似度 | `cosine` | 向量检索标准方案 |
| 认证 | basic auth（elastic / wsy2642@） | 同 Docker 配置 |
| SSL | 跳过证书校验 | 开发环境，ES 自签名证书 |

## 关键功能模块

### 1. ES 客户端初始化

- 从 `application.yml` 读取 host/port/username/password
- 跳过 SSL 证书校验（`SSLContext.getTrustAll()`）
- 启动时检测 ES 连通性

### 2. IndexManager（索引管理）

- 启动时自动创建索引，若已存在则跳过
- mapping 定义：

```json
{
  "mappings": {
    "properties": {
      "chunkId":    { "type": "keyword" },
      "md5":        { "type": "keyword" },
      "index":      { "type": "integer" },
      "text":       { "type": "text", "analyzer": "ik_max_word" },
      "charCount":  { "type": "integer" },
      "denseVector": { "type": "dense_vector", "dims": 2048, "index": true, "similarity": "cosine" }
    }
  }
}
```

### 3. VectorStoreConsumer

- 监听 `embedding.requested`，group-id = `localrag-vector-store`
- 反序列化 → 取 chunks → 逐条写入 ES
- 批量 `BulkRequest`，每批 50 条
- 失败重试，手动提交 offset

### 4. VectorStoreService

- 接口：`indexChunks(List<VectorChunk> chunks)`
- 实现：ES `bulk()` API 批量索引

## 边界条件

| 场景 | 处理策略 |
|------|----------|
| ES 不可用 | 消费者重试，消息不丢失 |
| 索引已存在 | 跳过创建 |
| 空 chunks | 打 warn，ACK 跳过 |
| 向量维度不匹配 | ES 写入时自动校验，不匹配报错 |
| 重复 chunkId | ES `_id = chunkId`，覆盖写入（幂等） |
| IK 分词器未安装 | mapping 创建失败，启动时报错 |

## 潜在技术难点

**1. ES 8.x 客户端 API（难度：中）**

`elasticsearch-java` 8.x 是新版 API，与旧版 `RestHighLevelClient` 完全不同。使用 Builder 模式 + `co.elastic.clients.elasticsearch` 包。需要熟悉新 API 的 `IndexRequest`、`BulkRequest` 构造方式。

**2. SSL 自签名证书（难度：低）**

开发环境 ES 开启 security 但使用自签名证书。客户端需配置 `SSLContext` 信任所有证书，或导入 CA 证书到 truststore。

> 当前用跳过校验方案，生产环境务必换成正式证书。

**3. dense_vector 维度校验（难度：低）**

ES 8.x 的 dense_vector 在 mapping 中声明固定维度。若传入向量维度不匹配（如通义千问 v3 默认 1024），写入会直接报错。嵌入模块已显式指定 2048，但需在第一个 chunk 写入后验证。

**4. IK 分词器可用性（难度：低）**

Docker 启动脚本已包含 IK 安装，但首次启动安装后需重启 ES。建议在 IndexManager 创建索引前检测 `_analyze` API 确认 IK 可用。

## 输入输出规范

### 输入（Kafka 消息）

```json
{
  "messageId": "uuid",
  "topic": "embedding.requested",
  "payload": {
    "md5": "d41d8cd...",
    "chunks": [
      {
        "chunkId": "d41d8cd_0001",
        "index": 1,
        "text": "第一章 概述...",
        "charCount": 3500,
        "denseVector": [0.012, -0.034, ...2048 floats]
      }
    ]
  }
}
```

### 输出

无 Kafka 输出（入链终点）。ES 中每条 chunk 对应一个 document。

## 测试验证点（入链完成）

1. 上传一个 txt 文件 → 检查 MinIO Console 是否有对象
2. 查看 Kafka 消息：`kafka-console-consumer --topic embedding.requested`
3. 查询 ES：`GET localrag-chunks/_search`
4. 验证字段：chunkId、text、denseVector 是否齐全
