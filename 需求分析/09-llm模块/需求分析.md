# LLM 模块 — 需求分析

## 核心业务逻辑

接收用户问题 → 调 retrieval 检索相关 chunk → 拼接 RAG Prompt（含历史对话前 15 条） → 调 DeepSeek Chat API → SSE 流式输出给前端。

### 数据流

```
用户提问 "什么是 RAG？"
  │
  │  1. retrievalService.search(query, topK=5) → List<RetrievalResult>
  │  2. 取前 15 条历史消息
  │  3. 拼装 RAG Prompt
  │     ├── 系统提示词
  │     ├── 参考资料（chunks）
  │     ├── 历史对话（最近 15 条）
  │     └── 当前问题
  │  4. DeepSeek API → SSE 流式输出
  ▼
前端逐字展示
```

### Prompt 模板

```
你是 LocalRAG 知识库助手。只根据【参考资料】回答，不要编造。

【参考资料】
{chunk1_text}
{chunk2_text}
...

【历史对话】
用户：xxx
助手：xxx
...

【用户问题】
{question}
```

### 关键决策

| 决策点 | 方案 | 理由 |
|--------|------|------|
| 模型 | **DeepSeek Chat（deepseek-chat）** | 用户已有 Key |
| 输出方式 | **SSE 流式** | 逐字返回，体验好 |
| 历史记录 | **保留最近 15 条** | 足够上下文，不超 token 上限 |
| 历史存储 | **Redis（当天热数据）+ MySQL（全量持久）** | 当天高频读写走 Redis，历史查询落 MySQL |
| 检索结果条数 | Top 5 个 chunk | 拼在 Prompt 里不超 token 限制 |
| 超时 | 60s | 大模型生成可能较慢 |

## 对话历史存储策略

### Redis + MySQL 双层架构

```
写操作（每次对话完成后）：
  用户消息 + 助手回复 → 同时写 Redis + MySQL

读操作（下次对话时取历史）：
  → 先查 Redis（当天数据，毫秒级）
  → 不足 15 条 → 补查 MySQL
  → 拼装返回
```

| 层 | 存储 | 用途 | 过期策略 |
|----|------|------|----------|
| Redis | `chat:{sessionId}` List | 当天热数据，高频读写 | TTL = 当天剩余秒数（每天 0 点过期） |
| MySQL | `chat_history` 表 | 全量持久，历史追溯 | 永久保留（后续可加归档策略） |

### 为什么用这个方案

- **Redis 存当天**：对话高峰期，同一条 session 可能几分钟内多次请求，Redis 亚毫秒响应，避免每次查 MySQL
- **MySQL 兜底**：跨天对话、历史回溯、审计需求，MySQL 是数据源
- **分层不是双写负担**：每次对话只写 2 条记录（user + assistant），写入量极小，双写成本可忽略
- **过期即清理**：Redis TTL 到当天 24:00，不占额外内存，第二天自动从 MySQL 加载

### 数据流示例

```
用户第二次打开对话（sessionId = "s1"）：
  1. Redis LRANGE chat:s1 -15 -1 → 返回 3 条（当天只聊了 3 轮）
  2. 3 < 15 → MySQL SELECT * FROM chat_history WHERE session_id='s1' ORDER BY created_at DESC LIMIT 15
  3. 合并去重 → 取最近 15 条 → 拼入 Prompt
```

### MySQL 表结构

```sql
CREATE TABLE chat_history (
    id          BIGINT AUTO_INCREMENT PRIMARY KEY,
    session_id  VARCHAR(64)  NOT NULL,
    role        VARCHAR(16)  NOT NULL,   -- 'user' | 'assistant'
    content     TEXT         NOT NULL,
    created_at  DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_session (session_id, created_at)
);
```

### Redis 数据结构

```
Key:   chat:{sessionId}
Type:  List
Value: JSON 字符串 [{role, content, timestamp}, ...]
TTL:   当天 23:59:59 - now()
操作:  RPUSH 追加, LRANGE 读取, EXPIRE 续期
```

## 关键功能模块

### 1. LlmService 接口

```
输入：String query, String sessionId
输出：SseEmitter（流式）
```

### 2. DeepSeekLlmService 实现

- 端点：`https://api.deepseek.com/chat/completions`
- 模型：`deepseek-chat`
- 认证：`Authorization: Bearer {apiKey}`
- 流式：`stream: true`
- 用 OkHttp 发请求，逐行读 SSE 响应

### 3. ChatHistoryManager（对话历史，双层存储）

```java
// 写入：同时写 Redis + MySQL
void append(String sessionId, String role, String content);

// 读取：先 Redis → 不足补 MySQL → 返回最近 15 条
List<Message> getRecent(String sessionId, int count);
```

- **Redis**：`RPUSH chat:{sessionId} {json}` → `LRANGE -15 -1` 快速取
- **MySQL**：`INSERT INTO chat_history` 持久化 → `SELECT ... LIMIT 15` 补查
- Redis TTL 设到当天 23:59:59，跨天自动过期
- 返回时自动按时间排序，取最近 15 条

### 4. PromptBuilder（Prompt 拼装）

```
系统提示词 → 参考资料 → 历史对话 → 当前问题
```

优先级：系统角色固定，参考资料从检索结果取，历史对话从 ChatHistoryManager 取。

## 边界条件

| 场景 | 处理策略 |
|------|----------|
| 检索返回空 | Prompt 提示"未找到相关信息"，不传参考资料 |
| 无历史对话 | 不拼接历史部分 |
| 历史超 15 条 | 取最近 15 条，旧的自然淘汰 |
| DeepSeek API 超时 | 60s 超时，返回错误 |
| 用户连续发问 | 异步追加到历史，不阻塞当前请求 |
| sessionId 为空 | 用 "default" 作为默认会话 |

## 潜在技术难点

**1. SSE 流式响应解析（难度：中）**

DeepSeek 返回的 SSE 格式：`data: {"choices":[{"delta":{"content":"你"}}]}`。用 OkHttp 逐行读取 `response.body().source()`，解析 `data:` 前缀的 JSON 行，提取 `delta.content` 写入 SseEmitter。

**2. Token 控制（难度：低）**

5 个 chunk × 每块 1500 字 ≈ 7500 字，加上系统提示词和历史对话，总计约 10K tokens。DeepSeek 支持 64K 上下文，足够。

**3. 连接中断处理（难度：低）**

用户关闭页面 → SseEmitter 的 `onCompletion` 回调触发 → 标记 SSE 连接关闭 → OkHttp 的 stream 也被取消。

## 输入输出规范

### API 端点

```
POST /api/llm/chat
Content-Type: application/json

Request:
{
  "query": "什么是 RAG？",
  "sessionId": "session-001"
}

Response: SSE 流式
data: {"content":"RAG","done":false}
data: {"content":" 是","done":false}
data: {"content":" 检索","done":false}
...
data: {"content":"","done":true}
```

### 前端 SSE 消费

```javascript
const eventSource = new EventSource('/api/llm/chat?query=...&sessionId=...');
eventSource.onmessage = (e) => {
    const { content, done } = JSON.parse(e.data);
    if (done) { eventSource.close(); return; }
    appendToChat(content);
};
```
