# Document 模块 — 需求分析

## 核心业务逻辑

消费 Kafka `document.uploaded` 消息，从 MinIO 下载文件，解析文本内容，分块，然后发送 `document.chunked` 消息给下一环节。

### 数据流

```
Kafka (document.uploaded)
  │
  │  FileUploadedPayload {md5, fileName, fileSize, objectKey}
  ▼
DocumentConsumer
  │
  │  1. 去重检查
  │  2. MinIO 下载文件
  │  3. Apache Tika 解析 → 纯文本
  │  4. TextChunker 流式分块（段落 → 句子 → HanLP 三级降级）
  │  5. 发布 document.chunked 消息
  ▼
Kafka (document.chunked)
  │
  │  DocumentChunkedPayload {md5, chunks: [{chunkId, text, charCount, denseVector}]}
  ▼
Embedding 模块 → ES 存储（dense_vector 2048 + text BM25）
```

### 关键决策

| 决策点 | 方案 | 理由 |
|--------|------|------|
| 解析引擎 | **Apache Tika** | 一个依赖覆盖 1000+ 格式 |
| 执行模式 | **消费 → 下载 → Tika 解析 → 分块 → 发送，顺序链** | 每步依赖上一步，无并发复杂度 |
| 分块策略 | **段落 → 句子 → HanLP 分词，三级降级** | 优先保语义完整性，实在不行用分词兜底 |
| 流式缓冲 | **Tika 输出累积到 5000 字符开始切块** | 避免全文档加载到内存 |
| 重叠 | **15% 重叠** | 避免跨块语义断裂，比固定值更灵活 |
| 向量维度 | **2048** | 通义千问 text-embedding-v3 输出维度 |
| 检索方式 | **KNN 向量检索 → BM25 混合排序** | 语义 + 关键词互补 |
| 去重 | 内存 Set<String md5> | 当前阶段够用 |

### 分块策略详解

```
Tika 解析输出（流式文本）
  │
  ├─ 累积到 buffer，buffer.length >= 5000 时开始处理
  │
  ├─ 1. 按段落分割（\n\n）
  │     │
  │     ├─ 段落 < 512 字符 → 合并到当前块
  │     │
  │     └─ 段落 >= 512 字符 → 按句子分割（。！？；）
  │           │
  │           ├─ 句子 < 512 → 作为独立块
  │           │
  │           └─ 句子 >= 512 → HanLP 分词后按 token 切分
  │
  ├─ 2. 相邻块之间保留 15% 重叠
  │     chunk_i 末尾 15% 的内容 = chunk_i+1 开头的 15%
  │
  └─ 3. 输出 List<Chunk>
```

**重叠计算**：`overlap = min(chunkText.length * 15%, 512)`，即重叠不超过 512 字符。

## 关键功能模块

### 1. Kafka 消费者（DocumentConsumer）

- 监听 `document.uploaded`，group-id = `localrag-document`
- 反序列化 `KafkaMessage<FileUploadedPayload>` → 取出 payload
- 去重：已处理的 md5 跳过
- 调用解析 → 分块 → 发送 document.chunked
- 手动提交 offset，失败不提交

### 2. 文档解析（Apache Tika）

- 单一依赖 `tika-parsers-standard-package`，自动识别格式
- `TikaDocumentParser.parse(InputStream)` → 纯文本 String
- 自动处理编码、字体、元数据剥离
- 不支持的格式返回空文本，打 warn 日志

### 3. 文本分块器（TextChunker）

**流式处理**：Tika 解析输出累积到 buffer，达到 5000 字符触发分块处理。

**三级降级切分**：

```
段落分割（\n\n）
  → 段落 < 512 → 直接入块
  → 段落 >= 512 → 句子分割（。！？；\n）
      → 句子 < 512 → 独立块
      → 句子 >= 512 → HanLP 分词切分
```

**重叠**：相邻块保留 15% 重叠，`overlapSize = min(chunkText.len * 15%, 512)`

**输出**：`List<Chunk>`，每个 Chunk 含 `{chunkId, index, text, charCount}`

**依赖**：`com.hankcs:hanlp:portable-1.8.4`

### 4. 消息生产者（复用 messaging 模块）

- 分块完成后发送 `document.chunked`
- Payload：`DocumentChunkedPayload {md5, fileName, chunks: [{chunkId, index, text, charCount}]}`

### 5. 顺序执行链（DocumentConsumer 入口）

```
onMessage → 下载 → Tika 解析 → TextChunker 分块 → 发送 document.chunked → ACK
```

每步依赖上一步结果，同步顺序执行。任一步失败打日志 + 不提交 offset，消息不丢失。

## 边界条件

| 场景 | 处理策略 |
|------|----------|
| 不支持的格式 | Tika 返回空或乱码 → 打 warn，标记已处理 |
| 空文件 | 解析出空文本 → 不分块 → 标记已处理 |
| PDF 扫描件（无文字） | Tika 返回空 → 打日志，标记已处理 |
| 编码异常 | Tika 自动检测编码，兜底 UTF-8 |
| 重复消息 | md5 去重，已处理直接 ACK |
| 解析异常 | 打 error 日志，不提交 offset |
| 单段落极长 | HanLP 分词后按 token 强制切分 |
| 分块数 > 1000 | 增大 buffer 阈值（5000 → 8000），合并小块 |

## 潜在技术难点

**1. Tika 依赖体积（难度：低）**

`tika-parsers-standard-package` 约 60MB，首次下载较慢。用 `tika-core` + 按需引入 `tika-parser-*` 可减小体积，但当前阶段直接用 standard-package 最简单。

**2. 中文分块边界（难度：中）**

英文可天然按空格分词，中文没有分隔符。当前按字符数切分 + 段落边界对齐。后续可优化为按句子拆分（调用嵌入模型 tokenizer）。

**3. 大文件内存控制（难度：低）**

Tika 解析结果全部加载到内存。单文件文本内容通常远小于原始文件（去掉格式后），50MB PDF 提取文本约 1-5MB，可控。超大文件（>500MB）建议在 storage 层面拦截。

**4. 顺序执行阻塞（难度：低）**

同步链式执行，单条消息处理耗时约 1-5 秒（下载 + 解析 + 分块）。当前文件量不大时完全够用，后续若需要并发处理，改为线程池 + 批量提交 offset。

## 输入输出规范

### 输入（Kafka 消息）

```json
{
  "messageId": "uuid",
  "topic": "document.uploaded",
  "timestamp": "2026-05-05T18:00:00Z",
  "payload": {
    "md5": "d41d8cd...",
    "fileName": "项目文档.pdf",
    "fileSize": 104857600,
    "objectKey": "d41d8cd.../项目文档.pdf"
  }
}
```

### 输出（Kafka 消息）

```json
{
  "messageId": "uuid",
  "topic": "document.chunked",
  "timestamp": "2026-05-05T18:01:00Z",
  "payload": {
    "md5": "d41d8cd...",
    "fileName": "项目文档.pdf",
    "chunks": [
      {
        "chunkId": "d41d8cd_0001",
        "index": 1,
        "text": "第一章 概述...",
        "charCount": 1450
      },
      {
        "chunkId": "d41d8cd_0002",
        "index": 2,
        "text": "第二章 架构设计...",
        "charCount": 1520
      }
    ]
  }
}
```

### 分块参数

| 参数 | 值 | 说明 |
|------|-----|------|
| bufferSize | 5000 字符 | 累积到此阈值触发切块 |
| minChunk | 512 字符 | 低于此值的段落合并到相邻块 |
| overlap | 15% | 相邻块重叠比例，最大不超过 512 字符 |
| 向量维度 | 2048 | denseVector 字段维度（embedding 模块写入） |

### Chunk 存储字段（ES）

| 字段 | 类型 | 说明 |
|------|------|------|
| chunkId | keyword | `{md5}_{index}` |
| md5 | keyword | 所属文件 |
| text | text | 切块内容（BM25 索引） |
| charCount | integer | 字符数 |
| denseVector | dense_vector(2048) | 嵌入向量（embedding 模块生成） |

### 检索链路（后续 vector-store + retrieval 模块）

```
用户查询 → 向量化(2048d)
  → ES KNN 向量召回 Top 200
     + BM25 关键词召回 Top 200
  → 合并去重
  → Rerank 精排 → Top 10
  → 返回给 LLM
```
