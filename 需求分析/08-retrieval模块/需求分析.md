# Retrieval 模块 — 需求分析

## 核心业务逻辑

接收用户查询文本，调 embedding 服务将其转为向量，在 ES 中执行**混合检索**（KNN 向量相似度 + BM25 关键词匹配），合并去重后**精排（Rerank）**，返回 Top-K 个最相关的 chunk 给 LLM 模块。

### 检索策略（三级瀑布）

```
用户查询 "什么是 RAG？"
  │
  │  1. embeddingService.embed(query) → float[1024]
  ▼
第一轮：KNN 向量检索
  │  ES dense_vector KNN → Top 30
  ▼
第二轮：BM25 关键词重排
  │  在 KNN 返回的 30 条结果上，用 BM25 重新打分排序
  ▼
第三轮：Rerank 精排 → Top 10
  │  对前 10-15 条用 rescore 精确排序
  ▼
返回 Top 10 → LLM 模块
```

### 关键决策

| 决策点 | 方案 | 理由 |
|--------|------|------|
| 第一轮 | **KNN 向量召回 Top 30** | 语义匹配优先，缩小候选集 |
| 第二轮 | **BM25 在 30 条内重排** | 关键词精确匹配，调整语义偏差 |
| 第三轮 | **rescore 精排 → Top 10** | 最终精确排序 |
| 向量相似度 | cosine | embedding 模型用 l2 normalize |
| 最终输出 | Top 5-10 个 chunk | 输入给 LLM 的上下文窗口有限 |

**为什么是 30 → BM25 → Rerank 而非并行**：先语义召回保证覆盖，再关键词校正排序，最后精排。每步收窄候选集，避免分数量纲不一致的问题。

## 关键功能模块

### 1. RetrievalService 接口

```
输入：String query, int topK
输出：List<RetrievalResult>
  - chunkId, md5, text, charCount, score, denseVector
```

### 2. ES 混合检索策略

```json
{
  "query": {
    "bool": {
      "should": [
        {
          "knn": {
            "field": "denseVector",
            "query_vector": [0.012, -0.034, ...],
            "k": 200,
            "num_candidates": 500
          }
        },
        {
          "match": {
            "text": {
              "query": "什么是 RAG？",
              "boost": 0.5
            }
          }
        }
      ]
    }
  },
  "size": 200
}
```

### 3. Rerank 精排

ES 8.10 原生支持 `rescore`，对粗排结果用更精确的算法重排序：

```json
{
  "rescore": {
    "window_size": 50,
    "query": {
      "rescore_query": {
        "match": {"text": "什么是 RAG？"}
      },
      "query_weight": 0.3,
      "rescore_query_weight": 0.7
    }
  }
}
```

先粗排召回 200 → 对前 50 条精排 → 返回 Top 10。

### 4. RetrievalConsumer（可选，后续加）

如果检索需要异步处理（如复杂的 Rerank 模型），可以用 Kafka 解耦。当前阶段同步调用即可。

## 边界条件

| 场景 | 处理策略 |
|------|----------|
| ES 返回空结果 | 返回空列表，LLM 走兜底："未找到相关信息" |
| query 为空或纯空格 | 拒绝检索，返回空 |
| ES 超时/不可用 | 抛 RetrievalException，API 层返回 500 |
| KNN 字段不存在 | ES 报错，启动时 IndexManager 保证索引已建 |
| 去重后不足 TopK | 有多少返回多少 |

## 潜在技术难点

**1. KNN + BM25 分数融合（难度：中）**

KNN 返回的是 cosine 相似度（0~1），BM25 返回的是词频分数（无上限）。直接 merge 后按分数排序会有偏差。解决：两路各自 Top 200，通过 `doc_id` 去重，在 Rerank 阶段用统一的 rescore 重新打分，避免分数量纲不一致。

**2. denseVector 查询格式（难度：低）**

ES 8.10 的 KNN 查询使用 `knn` 子句，需要在 `query_vector` 中传入 float 数组。embedding 模块输出的 `float[]` 直接可用，无需额外处理。

**3. 索引名硬编码（难度：低）**

检索时需指定索引名 `localrag-chunks`，与 vector-store 模块保持一致。
