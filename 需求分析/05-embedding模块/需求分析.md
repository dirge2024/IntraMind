# Embedding 模块 — 需求分析

## 核心业务逻辑

消费 Kafka `document.chunked` 消息，调用嵌入模型 API（通义千问/DeepSeek）将每个 chunk 文本转为 2048 维向量，然后发送 `embedding.requested` 消息给 vector-store 模块。

### 数据流

```
Kafka (document.chunked)
  │
  │  DocumentChunkedPayload {md5, fileName, chunks: [{chunkId, text, charCount}]}
  ▼
EmbeddingConsumer
  │
  │  1. 批量取出 chunks
  │  2. 调 EmbeddingService.embed(texts) → List<float[2048]>
  │  3. 每个 chunk 挂上 denseVector
  │  4. 发送 embedding.requested 消息
  ▼
Kafka (embedding.requested)
  │
  │  EmbeddingRequestedPayload {md5, chunks: [{chunkId, text, denseVector}]}
  ▼
Vector-Store 模块（写入 ES）
```

### 关键决策

| 决策点 | 方案 | 理由 |
|--------|------|------|
| 接口抽象 | `EmbeddingService` 接口 + 双实现 | 自由切换模型，不耦合调用方 |
| 批量调用 | 一次 API 请求嵌入多条 text | 减少网络往返，提升吞吐 |
| 向量维度 | 2048 | 业务指定，v3 原生支持 |
| 单次批量上限 | 25 条 | API 限制 + token 上限 |
| 文本截断 | 超过 8000 字符截断前 8000 | 通义千问 v3 max_tokens ≈ 8192 |
| 失败重试 | 3 次指数退避（1s/2s/4s） | API 偶发抖动 |
| 模型切换 | `application.yml` 配置 `model: qwen \| deepseek` | Spring @ConditionalOnProperty |
| HTTP 客户端 | OkHttp | 轻量，已在 dependencyManagement 中 |

## 关键功能模块

### 1. EmbeddingService 接口

```
输入：List<String> texts
输出：List<float[]> vectors（每个 float[2048]）
```

### 2. QwenEmbeddingService

- 端点：`https://dashscope.aliyuncs.com/api/v1/services/embeddings/text-embedding/text-embedding`
- 模型：`text-embedding-v3`
- 认证：Header `Authorization: Bearer {apiKey}`
- 维度：`dimensions: 2048`
- 请求体：
```json
{
  "model": "text-embedding-v3",
  "input": { "texts": ["chunk1", "chunk2"] },
  "parameters": { "dimension": 2048 }
}
```

### 3. DeepSeekEmbeddingService

- 端点：`https://api.deepseek.com/v1/embeddings`
- 模型：`deepseek-embedding`
- 认证：Header `Authorization: Bearer {apiKey}`
- 请求体（OpenAI 兼容）：
```json
{
  "model": "deepseek-embedding",
  "input": ["chunk1", "chunk2"]
}
```

### 4. EmbeddingConsumer

- 监听 `document.chunked`，group-id = `localrag-embedding`
- 反序列化 → 取 chunks → 分批（25 条/批）
- 调 `embeddingService.embed(batch)` → 获取向量
- 每个 chunk 挂上 `denseVector`
- 发送 `embedding.requested` 消息
- 失败重试，手动提交 offset

### 5. 消息 Payload（EmbeddingRequestedPayload，放 common）

```
{md5, chunks: [{chunkId, text, charCount, denseVector: float[2048]}]}
```

## 边界条件

| 场景 | 处理策略 |
|------|----------|
| API Key 未配置 | 启动时报错，拒绝启动 |
| API 超时 | 30s 超时，重试 3 次 |
| API 返回异常 | 指数退避重试，3 次后跳过该批次 |
| 单文本超长 | 截断前 8000 字符，保留尾部语义 |
| batch 大小不整除 | 最后一批不足 25 条直接发 |
| chunks 数量为 0 | 打 warn 日志，ACK 跳过 |
| 模型切换 | yml 改一个值，无需动代码 |

## 潜在技术难点

**1. 通义千问 API 格式差异（难度：低）**

通义千问的 embedding API 入参格式为 `{model, input: {texts: [...]}}`，与 OpenAI 的 `{model, input: [...]}` 不同。两个实现各自适配，上层无感。

**2. 向量维度一致性（难度：低）**

通义千问 v3 默认 1024 维，需显式传 `parameters.dimension=2048`。DeepSeek 输出维度固定，需确认是否 2048。

**3. 大批量 chunks 耗时（难度：中）**

100 个块 = 4 批 API 调用，每批 ~500ms，总耗时 ~2s。当前单线程消费可接受，后续可改为异步 + 批量 offset 提交。

**4. API 费用控制（难度：低）**

每批 25 条 × 8000 字符 ≈ 200K tokens。记录每次调用的 token 消耗日志，便于后续成本统计。

## 输入输出规范

### 输入（Kafka 消息）

```json
{
  "messageId": "uuid",
  "topic": "document.chunked",
  "payload": {
    "md5": "d41d8cd...",
    "fileName": "项目文档.pdf",
    "chunks": [
      {"chunkId": "d41d8cd_0001", "index": 1, "text": "第一章...", "charCount": 3500},
      {"chunkId": "d41d8cd_0002", "index": 2, "text": "第二章...", "charCount": 4200}
    ]
  }
}
```

### 输出（Kafka 消息）

```json
{
  "messageId": "uuid",
  "topic": "embedding.requested",
  "payload": {
    "md5": "d41d8cd...",
    "chunks": [
      {
        "chunkId": "d41d8cd_0001",
        "index": 1,
        "text": "第一章...",
        "charCount": 3500,
        "denseVector": [0.012, -0.034, ...]  // 2048 floats
      }
    ]
  }
}
```
