### 1. MinIO 8.5.7 API 方法与预期不匹配 + interface 包名冲突

**时间**：2026-05-05

**遇到的问题**：
- 在 StorageServiceImpl 中使用了 `createMultipartUpload`、`uploadPart`、`listParts`、`completeMultipartUpload`、`abortMultipartUpload` 等 S3 风格方法，编译报错：方法不存在
- 同时 `com.localrag.storage.interface` 包名中的 `interface` 是 Java 保留字，IDE 解析为关键字导致连锁报错

**排查方式**：
1. 先确认 minio 8.5.7 jar 位置：`C:\Users\26424\.m2\repository\io\minio\minio\8.5.7\minio-8.5.7.jar`
2. 用 `javap -public` 反编译 MinioClient 类，列出所有公开方法
3. 发现 8.5.7 版本已全面迁移至 Builder 模式，旧式 S3 multipart 方法全部移除，只保留 `composeObject`（服务端合并）用于类似场景
4. 确认 `interface` 包名冲突后对所有模块做全局替换

**解决方法**：
1. 重写 StorageServiceImpl，放弃 S3 multipart API，改用：
   - 每个分片作为独立对象存储：`{md5}/chunks/{partNumber}`
   - 上传分片用 `putObject()`
   - 合并分片用 `composeObject()`（MinIO 服务端合并，不消耗应用流量）
   - 合并成功后删除临时 chunk 对象
2. 所有模块 `interface` 包重命名为 `contract`（`sed` 批量替换）

**教训**：
- 引入新依赖前先 `javap` 确认版本对应的实际 API，不要凭经验假设
- Java 项目避免用 `interface` 这类保留字做包名，用 `contract` 或 `api` 代替

### 2. Spring Boot 首次启动连续失败

**时间**：2026-05-05

**遇到的问题**：
1. `mvn spring-boot:run -pl api` 报依赖找不到（common/storage 等模块 jar 不在本地仓库）
2. 修复后启动报 `Failed to bind properties under 'spring.web.resources'`（yml 中删掉 `static-locations` 后留下空 `resources:` 节点）
3. 端口 8080 被其他项目占用

**排查方式**：
1. 确认多模块项目需先 `mvn install` 将各模块 jar 安装到 `~/.m2`
2. 查看完整启动日志，定位到 `spring.web.resources` 绑定失败报错
3. 检查 `application.yml` 发现 `web.resources:` 下有子节点但值为空

**解决方法**：
1. `mvn clean install -DskipTests` 安装全部模块，再 `mvn spring-boot:run -pl api` 启动
2. 删除空的 `spring.web.resources` 节点（Spring Boot 默认从 `classpath:/static/` 提供静态资源）
3. 端口改为 8088

**教训**：
- Maven 多模块项目 `-pl` 只编译目标模块，依赖模块必须先 `install`
- 编辑 yml 时如需删除配置，连同父节点一起删干净，不能留下空 key
