# 开发问题记录

### 1. MinIO 8.5.7 版本 S3 风格 API 全部不可用

**时间**：2026-05-05

**现象**：在 StorageServiceImpl 中使用 `createMultipartUpload`、`uploadPart` 等 S3 标准方法，编译报错方法不存在。同时 `interface` 包名被 IDE 解析为 Java 关键字导致连锁报错。

**造成原因**：MinIO Java SDK 在 8.5.x 已全面废弃旧式 S3 风格方法，统一为 Builder 模式。凭经验假设 API 签名，没验证实际 jar 中的方法列表。

**解决方法**：
1. 用 `javap -public` 反编译 MinioClient.class 确认 8.5.7 的实际 API——只有 `putObject`、`composeObject` 等 Builder 方法
2. 改为 composeObject 方案：每个分片上传为独立对象 `{md5}/chunks/00001`，全部完成后调 `composeObject` 在 MinIO 服务端合并，再删除临时分片
3. 所有模块 `interface` 包名批量改为 `contract`

### 2. Maven 多模块首次启动失败

**时间**：2026-05-05

**现象**：`mvn spring-boot:run -pl api` 报所有内部模块 jar 找不到，修复后 Spring 容器初始化报 `spring.web.resources` 属性绑定失败，还有端口 8080 被占用。

**造成原因**：`-pl` 只指定编译目标模块，依赖解析仍走本地仓库 `~/.m2`。底下模块从未 `install` 过，本地仓库没有 jar。另外 YAML 中删了 `static-locations` 值但留下了空 key `spring.web.resources:`，Spring Boot 无法将其转换为 `WebProperties$Resources` 类型。

**解决方法**：先 `mvn clean install -DskipTests` 装所有模块到本地仓库，再 `spring-boot:run -pl api`。删除空的 YAML 节点。端口改为 8088。

### 3. ES 连接协议错误：HTTPS 连 HTTP 端口

**时间**：2026-05-05

**现象**：启动时 IndexManager 报 `VectorStoreException: ES 索引创建失败`，应用直接退出。

**造成原因**：Docker 的 ES 容器配置了 `xpack.security.http.ssl.enabled=false`（HTTP 明文 + basic auth），但 EsConfig 中写了 `new HttpHost(host, port, "https")`，客户端对 HTTP 端口做 SSL 握手被拒绝。

**解决方法**：改为 `"http"` 协议。IndexManager.run() 加 try-catch，ES 不可用时打 WARN 继续启动不阻塞。

### 4. Redis 脏数据：上次上传中断后残留不完整记录

**时间**：2026-05-05

**现象**：InitUpload 接口返回 500，后台 `NumberFormatException: For input string: ""`，在 `getByMd5()` 的 `Long.parseLong(fields.get("fileSize"))` 处。

**造成原因**：上一次 init 调用写入了 Redis Hash，但程序随后崩溃，留下字段不完整的记录。`getByMd5()` 只检查了 key 是否存在，没有逐字段校验值是否合法，空字符串直接传给了 `Long.parseLong`。

**解决方法**：`getByMd5()` 增加字段完整性校验——`fileSize`/`totalParts`/`status` 任一为空 → 视为脏数据 → 删除 Redis key → 返回 null 走新上传流程。外层加 try-catch 兜底。

### 5. Map.of() 空值崩溃：refactor 漏删一个字段引用

**时间**：2026-05-05

**现象**：InitUpload 再次 500，`NullPointerException` 堆栈指向 `ImmutableCollections$MapN.<init>`，由 `Map.of()` 触发。

**造成原因**：之前 refactor 把 MinIO 的 `createMultipartUpload` 从整个链路移除（改为 composeObject 方案），`uploadId` 不再有值。接口和 Controller 的赋值都删了，但 `tryCreate()` 里 `Map.of("uploadId", task.getUploadId())` 这一行漏了。`task.getUploadId()` 为 null，而 `Map.of()` 禁止 null 值，在构造时直接抛 NPE。

**解决方法**：`Map.of()` 改回 `new HashMap<>()` + `put()`，同时去掉 `uploadId` 字段的写入——composeObject 方案不需要它。

```java
// 改前：Map.of() 不接 null
Map.of("uploadId", task.getUploadId());  // NPE

// 改后：HashMap 允许 null，且不写 uploadId
Map<String, String> fields = new HashMap<>();
fields.put("md5", task.getMd5());
fields.put("bucket", task.getBucket());
```

**收获**：refactor 之后要全局搜索被删字段的引用，不能只改显式赋值的地方。`Map.of()`/`List.of()` 仅适合编译期常量场景，value 来自对象字段时用传统 HashMap。
